# Knowledge Vault 运行逻辑

> 本文档定义了 knowledge-vault 的整体运行逻辑，用于规范模型行为。
> **所有操作前必须完整阅读本文档。** 本文档是最高优先级的规则文件，优先级高于 .claude/skills/ 下的操作手册。

---

## 一、核心理念

### 1.1 知识是复利增长的

每次摄入新资料，LLM 将其编译进 Wiki，而不是每次查询都从头推导。随着时间的推移，知识库会越来越丰富，回答越来越精准。

**核心原则：** 每次操作都应该让知识库变得更好。不是完成一个任务就结束，而是让知识在库中沉淀、关联、复用。

**具体体现：**
- 一次 Ingest 产生的 Wiki 页面，会被后续的 Query 和 Synthesize 反复使用
- 一次 Synthesize 产生的综合报告，本身又成为后续操作的知识来源
- 一次 Lint 修复的问题，避免后续操作被错误信息误导
- 好的问答结果归档后，下次同样的问题可以直接引用，不需要重新推导

### 1.2 三层架构

```
┌──────────────────────────────────────┐
│  Raw Sources (只读, 人工维护)         │
│  原始资料：文章、论文、数据、转录...   │
├──────────────────────────────────────┤
│  The Wiki (LLM 维护, 人类阅读)        │
│  结构化知识：概念、实体、标准、综述...  │
├──────────────────────────────────────┤
│  The Schema (规则文件)                │
│  告诉 LLM 如何运作的规则              │
└──────────────────────────────────────┘
```

- **Raw 层**：原始资料，LLM 只读不写。用户放入后永不修改。
- **Wiki 层**：LLM 维护的知识库，面向人类阅读。内容结构化、可交叉引用。
- **Schema 层**：本文件 + CLAUDE.md + .claude/skills/ 下的操作手册。本文件最高优先级。

**为什么分层：**
- Raw 层保留原始证据，防止 LLM 在编译过程中丢失或扭曲信息
- Wiki 层是 LLM 编译后的结构化知识，便于人类阅读和 LLM 后续查询
- Schema 层约束 LLM 行为，确保操作一致性和质量

### 1.3 四大操作

| 操作 | 触发词 | 核心目标 | 产出 |
|------|--------|---------|------|
| **Ingest** | "保存到知识库"、"ingest"、"摄入"、"add to knowledge base"、"save to wiki" | 将新资料编译进 Wiki | raw/ 归档 + 多个 Wiki 页面 |
| **Query** | "查一下 X"、"知识库里"、"according to the knowledge base"、"what does the knowledge base say"、"wiki 里" | 基于 Wiki 回答问题 | 回答文本（可选归档） |
| **Synthesize** | "写一篇关于 X 的综述"、"综合报告"、"synthesize"、"pull together everything about X"、"综述" | 跨来源综合生成报告 | wiki/syntheses/ 报告 |
| **Lint** | "检查知识库"、"lint"、"健康检查"、"health check"、"clean up the knowledge base" | 发现并修复知识库问题 | 诊断报告 + 修复 |

## 二、目录结构逻辑

### 2.1 顶层结构

```
knowledge-vault/
├── README.md                # 项目说明（人类阅读）
├── CLAUDE.md                # 工作流规范（LLM 阅读）
├── VAULT-INDEX.md           # 实时仪表盘（LLM 自动维护）
├── KNOWLEDGE-VAULT-LOGIC.md # 本文件 — 运行逻辑
│
├── raw/                     # 原始资料层（只读，人工放入）
│   ├── laws/                # 法律法规层
│   ├── national-standards/  # 国家标准层
│   └── industry-standards/  # 行业标准层
│
├── wiki/                    # 知识库层（LLM 维护）
│   ├── index.md             # 主目录 — 从这里开始查询
│   ├── log.md               # 操作时间线（仅追加）
│   ├── hot.md               # 当前会话热缓存
│   ├── sources/             # 每个资料源的摘要页
│   ├── concepts/            # 概念解释页
│   ├── entities/            # 人物/公司/产品/机构页
│   ├── comparisons/         # 多维对比分析页
│   ├── syntheses/           # 跨源综合报告页
│   └── standards/           # 标准技术指标（按监管目录分类）
│
└── .claude/
    └── skills/              # 四个核心操作手册
        ├── ingest-wiki.md
        ├── query-wiki.md
        ├── synthesize-wiki.md
        └── lint-wiki.md
```

### 2.2 各层职责

#### raw/ 层（原始资料）

| 规则       | 说明                                                                                                               |
| -------- | ---------------------------------------------------------------------------------------------------------------- |
| **只读**   | LLM 永不修改 raw/ 下的原始文件。即使发现错误，也不修改，只在 Wiki 中标注                                                                     |
| **人工放入** | 用户下载/复制原始资料到此目录。LLM 不负责获取原始资料                                                                                    |
| **分类存放** | 按法规层级放入对应子目录：`laws/`（法律法规）、`national-standards/`（国家标准）、`industry-standards/`（行业标准）。不确定层级时放 `national-standards/` |
| **文件命名** | 以导入的标准编号和名称命名，如 `GB175—2023 通用硅酸盐水泥.md`。标准编号中的 `/` 替换为 `-`（如 `GB-T 15063-2020 复合肥料.md`）。保持与标准原文一致的名称 |
| **图片处理** | 如果资料包含图片，提示用户下载到 `raw/assets/`。LLM 不下载图片                                                                         |
| **完整性**  | 保持原始内容完整，不删减、不修改、不重新排版。即使是 OCR 错误也保留原样                                                                           |

#### wiki/ 层（知识库）

| 规则 | 说明 |
|------|------|
| **LLM 维护** | LLM 负责创建、更新、链接所有页面。用户不直接编辑 |
| **人类可读** | 内容面向人类阅读，清晰结构化。使用标题、列表、表格等 Markdown 元素 |
| **交叉引用** | 页面间通过 `[[wiki/page]]` 格式链接。引用必须指向存在的页面 |
| **双向维护** | A 引用 B 时，B 也应有回链。每次更新页面时检查相关页面的引用 |
| **YAML frontmatter** | 每个页面必须有 YAML frontmatter，包含 type、title、tags 等字段 |
| **一致性** | 同一类型的页面保持格式一致。使用本节提供的模板 |

### 2.3 standards/ 子目录的特殊结构

`wiki/standards/` 是本知识库的特色模块，采用 **5 层知识图谱** 结构：

```
L0: 产品类别（监管目录分类）
    例如：工业生产资料、农业生产资料、食品相关产品...
    │
L1: 标准节点（具体标准）
    例如：GB 175-2023、GB-T 15063-2020...
    │
L2: 指标类别（技术指标大类）
    例如：化学成分、物理性能、养分含量...
    │
L3: 指标子类（具体指标项）
    例如：初凝时间、抗压强度、水分(H₂O)...
    │
L4: 指标值（具体数值+约束条件）
    例如：通用 ≥45min、强度等级42.5 ≥42.5MPa...
```

**层级关系：**
- L1 标准节点包含多个 L2 指标类别
- L2 指标类别包含多个 L3 指标子类
- L3 指标子类包含多个 L4 指标值
- L4 指标值通过 YAML frontmatter 记录：约束条件（≥/≤）、数值、单位、等级/条件

**上基层链接规则（核心约束）：**
- **除 L0 外，每一层节点必须有指向上一层的链接，不得存在孤立节点**
- L0 产品类别 → 独立页面，无上基层，但需在 `wiki/index.md` 中登记
- L1 标准节点 → 通过目录归属（所在目录即为 L0 产品类别）
- L2 指标类别 → 页面中必须包含 `所属标准：[[wiki/standards/.../L1节点.md]]`
- L3 指标子类 → 页面中必须包含 `指标类别：[[wiki/standards/.../L2节点.md]]`
- L4 指标值 → 页面中必须包含 `子类：[[wiki/standards/.../L3节点.md]]`
- 新建标准时，必须从 L1 到 L4 完整创建，不允许只创建部分层级
- Lint 检查时，孤立节点（无上基层链接的节点）列为 P1 问题

**L0 产品类别节点规则：**
- 每个产品类别在 `wiki/standards/` 下有一个独立页面，文件名如 `工业生产资料.md`
- 即使该类别下暂时没有 L1-L4 标准，也先创建 L0 页面，后续逐步补充
- L0 页面内容：产品类别名称、所属监管目录说明、已有标准列表（随补充逐步更新）
- L0 页面在 `wiki/index.md` 的 standards/ 目录中登记

**文件命名约定：**

| 层级      | 命名规则                 | 示例                        |
| ------- | -------------------- | ------------------------- |
| L0 产品类别 | 监管目录名称（目录名）          | 工业生产资料/                   |
| L1 标准节点 | `GB 编号 标准名称.md`      | `GB 175-2023  通用硅酸盐水泥.md` |
| L2 指标类别 | `指标名称.md`            | `化学成分.md`                 |
| L3 指标子类 | `指标名称.md`            | `初凝时间.md`                 |
| L4 指标值  | `等级/条件 约束条件 数值单位.md` | `通用 ≥45min.md`            |

**为什么这么设计：**
- 每个指标值是一个独立文件，方便被多个标准共享引用（例如"通用 ≥45min"可能被多个水泥标准引用）
- 5 层结构支持精确查询：从产品类别→标准→指标类别→指标子类→具体数值
- YAML frontmatter 使指标值可被程序化查询（按约束条件、数值范围、单位筛选）
- 文件命名即标识，看到文件名就知道内容

## 三、操作流程逻辑（详细版）

### 3.1 Ingest（摄入）

**触发条件：** 用户说"保存到知识库"、"ingest"、"摄入"、"add to knowledge base"、"save to wiki"，或发送文件/内容并提到知识库。

#### 完整流程

```
Step 1: 判断资料类型
    ↓
Step 2: 归档原始资料到 raw/
    ↓
Step 3: 完整阅读并提取关键信息
    ↓
Step 4: 与用户讨论要点（可选，但推荐）
    ↓
Step 5: 更新 Wiki 页面
    ├── 创建/更新 wiki/sources/ 摘要页（必须）
    ├── 创建/更新 wiki/concepts/ 概念页（如有新概念）
    ├── 创建/更新 wiki/entities/ 实体页（如有新实体）
    ├── 创建 wiki/comparisons/ 对比页（如与已有知识形成对比）
    ├── 更新 wiki/index.md 目录（必须）
    └── 追加 wiki/log.md 操作记录（必须）
    ↓
Step 6: Git 提交
```

#### Step 1 — 判断资料层级

| 层级 | 判断依据 | raw/ 子目录 |
|------|---------|------------|
| 法律法规 | 法律、法规、条例、办法、实施细则 | raw/laws/ |
| 国家标准 | GB、GB/T、GB/Z 等国家标准全文或摘要 | raw/national-standards/ |
| 行业标准 | 行业标准（如 JT、QB、HG 等） | raw/industry-standards/ |

**判断规则：**
- 根据资料的标准编号前缀判断层级（GB → national-standards，行业代号 → industry-standards）
- 法律法规类文件（无标准编号）→ laws/
- 不确定时默认为 national-standards/

#### Step 2 — 归档规则

**文件命名：**
- 格式：`YYYY-MM-DD-简短描述.md`
- 描述用英文或中文均可，但同一批资料风格一致
- 示例：`2026-06-22-pytorch-20-release.md`

**内容保存：**
- 保持原始内容完整，不删减、不修改
- 如果资料是网页，保存完整 HTML 转 Markdown 后的内容
- 如果资料是 PDF，保存提取的文本内容（提示用户 PDF 可能提取不完整）
- 如果资料包含图片，提示用户下载到 `raw/assets/`

**图片处理规则：**
- LLM 不下载图片
- 在 raw/ 文件中用 `![图片描述](assets/文件名)` 占位
- 提示用户手动下载图片到 `raw/assets/`

#### Step 3 — 提取关键信息

**必须提取的信息：**
1. **核心观点** — 资料要传达的主要论点或发现
2. **关键数据** — 重要的数字、统计、指标值
3. **新概念** — Wiki 中不存在的新术语、新方法
4. **新实体** — 资料中提到的关键人物、公司、产品、机构
5. **关联** — 与已有知识的联系（支持、补充、矛盾）
6. **来源信息** — 作者、发布时间、来源链接

**矛盾识别：**
- 如果新资料与已有知识冲突，记录矛盾点
- 不判断谁对谁错，只记录"资料 A 说 X，资料 B 说 Y"
- 在相关页面标注 `[!contradiction]`

#### Step 4 — 与用户讨论要点

**推荐但非必须。** 以下情况建议讨论：
- 资料内容复杂，不确定哪些是重点
- 资料与已有知识矛盾，需要用户判断
- 资料涉及多个领域，不确定归类方式
- 用户可能有特定的关注方向

**讨论方式：**
- 简要总结资料的核心内容
- 列出 2-3 个可能的关注方向
- 询问用户希望强调哪些方面

#### Step 5 — 更新 Wiki 的详细决策逻辑

##### sources/ 摘要页（必须创建）

**文件位置：** `wiki/sources/YYYY-MM-DD-简短描述.md`

**必须包含的内容：**
```markdown
---
title: 资料标题
type: source
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [资料类型标签]
---

# 资料标题

- **来源链接：** [URL]
- **作者：** [作者名]
- **发布时间：** YYYY-MM-DD
- **资料类型：** article/paper/repo/transcript/data

## 摘要

[2-3 句话概括资料核心内容]

## 关键要点

1. [要点一]
2. [要点二]
3. [要点三]
4. [要点四]
5. [要点五]

## 与知识库的关联

- 关联概念：[[wiki/concepts/概念页.md]]
- 关联实体：[[wiki/entities/实体页.md]]
- 补充/支持/矛盾：[[wiki/sources/已有来源.md]]
```

**创建规则：**
- 每个资料源创建一个 sources/ 页面
- 如果资料与已有来源高度相关，在"与知识库的关联"中引用
- 关键要点控制在 3-5 个

##### concepts/ 概念页（条件创建）

**创建条件（满足任一即可）：**
1. 资料中引入了 Wiki 中不存在的新术语/新概念
2. 资料对已有概念提供了重要的补充信息
3. 资料中的概念需要与相关概念区分（如"监督学习"vs"半监督学习"）

**不创建的条件：**
- 概念在 Wiki 中已有完整页面
- 概念过于基础/通用（如"数据"、"算法"这类词不需要单独建页）
- 概念仅在资料中顺带提及，不是核心内容

**文件位置：** `wiki/concepts/概念名.md`

**内容模板：**
```markdown
---
title: 概念名称
type: concept
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [标签]
---

# 概念名称

## 定义

[清晰的定义]

## 关键属性

- [属性一]
- [属性二]

## 与相关概念的关联

- [[wiki/concepts/相关概念.md]] — [关系说明]
- [[wiki/concepts/相关概念.md]] — [关系说明]

## 来源

- [[wiki/sources/来源页.md]]
```

**更新规则：**
- 如果概念已存在，检查新资料是否提供了新的信息
- 如果有新信息，更新概念页的"updated"字段和内容
- 在"来源"部分追加新的来源引用

##### entities/ 实体页（条件创建）

**创建条件（满足任一即可）：**
1. 资料中提到了重要的人物（行业专家、学者、官员）
2. 资料中提到了重要的公司/组织/机构
3. 资料中提到了重要的产品/项目

**不创建的条件：**
- 实体仅在资料中顺带提及，不是核心内容
- 实体过于通用（如"中国政府"这类不需要单独建页）
- 实体在 Wiki 中已有完整页面

**判断"重要"的标准：**
- 该实体在资料中反复出现
- 该实体是资料的核心主题
- 该实体在后续查询中可能被频繁引用

**文件位置：** `wiki/entities/实体名.md`

**内容模板：**
```markdown
---
title: 实体名称
type: entity
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [标签]
---

# 实体名称

**类型：** 人物/公司/产品/机构

## 简介

[2-3 句话介绍]

## 关键信息

- [信息一]
- [信息二]

## 相关来源

- [[wiki/sources/来源页.md]]
```

##### comparisons/ 对比页（条件创建）

**创建条件（必须同时满足）：**
1. 新资料与已有知识在某个维度上形成对比
2. 对比有实际价值（能帮助理解或决策）

**对比维度举例：**
- 方法 A vs 方法 B（不同技术路线）
- 观点 A vs 观点 B（不同学术立场）
- 时期 A vs 时期 B（不同时间的数据）
- 标准 A vs 标准 B（不同标准的差异）
- 地区 A vs 地区 B（不同地区的实践）

**不创建的条件：**
- 对比过于 trivial（如"A 和 B 名字不同"）
- 对比的双方信息量严重不对称
- 用户没有对比的需求

**文件位置：** `wiki/comparisons/YYYY-MM-DD-对比主题.md`

**内容模板：**
```markdown
---
title: 对比主题
type: comparison
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [标签]
---

# 对比主题

## 对比维度

| 维度 | 方 A | 方 B |
|------|------|------|
| [维度一] | ... | ... |
| [维度二] | ... | ... |

## 分析

[对比分析结论]

## 来源

- [[wiki/sources/来源A.md]]
- [[wiki/sources/来源B.md]]
```

##### index.md 更新规则

**必须更新的内容：**
- 新增的 sources/ 页面 → 添加到"最近摄入"或对应分类
- 新增的 concepts/ 页面 → 添加到"概念"列表
- 新增的 entities/ 页面 → 添加到"实体"列表
- 新增的 comparisons/ 页面 → 添加到"对比分析"列表
- 新增的 syntheses/ 页面 → 添加到"综合报告"列表
- 新增的 standards/ 页面 → 更新"标准"目录

**更新方式：**
- 在对应分类下追加新条目
- 保持列表有序（按时间倒序或按字母序）
- 不删除已有条目
- 如果分类不存在，创建新分类

##### log.md 追加规则

**格式：**
```
- YYYY-MM-DD HH:mm | [Ingest] 资料标题 | 更新了 X 个页面
```

**示例：**
```
- 2026-06-22 11:30 | [Ingest] PyTorch 2.0 发布说明 | 更新了 5 个页面
```

**规则：**
- 只追加，不修改已有记录
- 时间使用 24 小时制
- 页面数量包括新建和更新的所有页面

#### 一次摄入的影响范围

单个资料通常涉及 5-15 个 Wiki 页面。每次更新一个页面时，检查并更新所有相关的交叉引用。

**影响范围检查清单：**
- [ ] 新建了 sources/ 页面
- [ ] 新建/更新了 concepts/ 页面（如有新概念）
- [ ] 新建/更新了 entities/ 页面（如有新实体）
- [ ] 新建了 comparisons/ 页面（如形成对比）
- [ ] 更新了 index.md
- [ ] 更新了 log.md
- [ ] 所有新建页面都有入站引用
- [ ] 所有引用都指向存在的页面

#### 边界情况处理

| 情况 | 处理方式 |
|------|---------|
| **资料质量低、信息量少** | 仍然归档到 raw/，sources/ 摘要页注明"信息量有限，仅包含 X 个有效要点" |
| **资料与已有知识完全重复** | 归档到 raw/，在 sources/ 中标注"与 [[wiki/sources/已有来源]] 内容重复"，不创建新的 concepts/entities |
| **资料与已有知识矛盾** | 归档并创建 sources/，在相关页面标注 `[!contradiction]`，列出双方来源和具体矛盾点 |
| **用户只给了链接（未下载内容）** | 提示用户先下载原始内容到 raw/，不执行后续步骤 |
| **资料包含敏感/隐私信息** | 提示用户，不摄入到 Wiki。raw/ 也不保存 |
| **资料是外语** | 归档原文，sources/ 用中文写摘要。概念/实体名保留原文并附中文翻译 |
| **资料非常大（>100页）** | 先归档 raw/，然后分批阅读和提取。每次处理一个章节 |
| **用户同时给了多个资料** | 逐个处理。先归档所有 raw/，再逐个阅读和更新 Wiki |
| **资料格式无法解析** | 提示用户提供可解析的格式（Markdown/纯文本），已保存的 raw/ 保留原格式 |

### 3.2 Query（查询）

**触发条件：** 用户说"查一下 X"、"知识库里"、"according to the knowledge base"、"what does the knowledge base say"、"wiki 里"等。

#### 完整流程

```
Step 1: 定位相关页面
    ├── 先读 wiki/index.md（了解全貌）
    └── 再读 wiki/hot.md（了解近期上下文）
    ↓
Step 2: 深入阅读
    ├── 逐页阅读相关页面
    ├── 跟踪交叉引用
    └── 发现矛盾时标注 [!contradiction]
    ↓
Step 3: 综合回答
    ├── 多源综合
    └── 附带来源引用
    ↓
Step 4: 可选归档（有价值的回答）
    ├── 保存到 wiki/syntheses/
    ├── 更新 wiki/index.md
    └── 更新 wiki/log.md
```

#### Step 1 — 定位相关页面

**阅读顺序：**
1. **先读 `wiki/index.md`** — 了解知识库全貌，找到可能相关的分类
2. **再读 `wiki/hot.md`** — 了解近期会话上下文，避免重复工作
3. **根据 index.md 的指引，定位具体页面**

**定位策略：**
- 如果问题是关于某个标准的 → 直接去 `wiki/standards/` 对应目录
- 如果问题是关于某个概念的 → 去 `wiki/concepts/` 查找
- 如果问题是关于某个实体的 → 去 `wiki/entities/` 查找
- 如果问题涉及多个方面 → 读 index.md 确定范围

#### Step 2 — 深入阅读

**阅读规则：**
- 逐页阅读相关页面，不要跳读
- 跟踪页面中的交叉引用链接，阅读被引用的页面
- 如果发现矛盾，记录矛盾点

**矛盾处理：**
- 发现矛盾时，不修改页面（那是 Lint 的工作）
- 在回答中标注 `[!contradiction]`，列出矛盾双方
- 示例：`[!contradiction] 页面 A 说 X，页面 B 说 Y，两者矛盾`

#### Step 3 — 综合回答

**回答格式决策：**

| 问题类型 | 判断依据 | 回答格式 |
|---------|---------|---------|
| 事实查询 | "X 是多少"、"X 的标准值是什么" | 直接回答 + 引用来源 |
| 对比查询 | "X 和 Y 有什么区别"、"哪个更好" | 表格形式 |
| 分析查询 | "X 的趋势是什么"、"为什么 X 会这样" | 结构化 Markdown 报告 |
| 综合查询 | "关于 X 的所有信息"、"全面介绍 X" | 多维度结构化输出 |

**引用格式：**
- 每个事实/数据后附上来源引用
- 引用格式：`（来源：[[wiki/sources/来源页.md]]）`
- 如果多个来源支持同一事实，列出所有来源

**回答结构：**
```
[直接回答/结论]

[详细说明]

[来源引用]
- [[wiki/sources/来源1.md]]
- [[wiki/sources/来源2.md]]

[知识不足说明（如有）]
以下内容在 Wiki 中未找到：[具体缺失内容]
```

#### Step 4 — 归档决策

**什么情况下应该归档：**
- 回答本身具有长期价值（如对比分析、发现关联、综合报告）
- 回答涉及多个来源的综合
- 回答填补了知识空白
- 用户明确说"保存这个回答"

**什么情况下不应该归档：**
- 简单的单源事实查询
- 临时性的上下文回答
- 用户明确说不需要保存
- 回答内容与已有综合报告高度重复

**归档方式：**
- 保存到 `wiki/syntheses/YYYY-MM-DD-主题.md`
- 更新 `wiki/index.md` 添加新条目
- 更新 `wiki/log.md`

#### 边界情况处理

| 情况 | 处理方式 |
|------|---------|
| **Wiki 中有完整答案** | 直接回答，引用来源。不需要额外说明 |
| **Wiki 中有部分答案** | 回答已知部分，明确标注"以下信息在 Wiki 中未找到：[具体内容]" |
| **Wiki 中完全没有** | 明确告知"知识库中未找到关于 X 的信息"，建议用户补充来源 |
| **多个页面给出矛盾信息** | 列出所有矛盾来源，标注 `[!contradiction]`，不做主观判断 |
| **用户问的是训练数据中的知识** | 优先使用 Wiki 内容。如果 Wiki 没有，告知"知识库中未收录" |
| **用户问的是 standards/ 中的指标值** | 按 5 层结构逐层定位：产品类别→标准→指标类别→指标子类→指标值 |
| **用户问的问题太宽泛** | 缩小范围：先问用户具体想了解哪个方面 |
| **用户要求实时数据** | 告知知识库不是实时更新的，建议查询最新官方来源 |

### 3.3 Synthesize（综合）

**触发条件：** 用户说"写一篇关于 X 的综述"、"综合报告"、"synthesize"、"pull together everything about X"、"综述"等。

#### 完整流程

```
Step 1: 扫描相关页面
    ├── 读 wiki/index.md 确定范围
    ├── 读所有相关页面（sources, concepts, entities, comparisons）
    └── 跟踪交叉引用，确保覆盖全面
    ↓
Step 2: 综合分析
    ├── 识别多源共识 vs 单一来源观点
    ├── 标注矛盾点 [!contradiction]
    └── 发现知识空白（Wiki 中缺失但应该有的内容）
    ↓
Step 3: 撰写报告
    ├── 保存到 wiki/syntheses/YYYY-MM-DD-主题.md
    └── 报告结构见下方
    ↓
Step 4: 更新 index.md + log.md
    ↓
Step 5: Git 提交
```

#### Step 1 — 扫描相关页面

**扫描规则：**
1. 先读 `wiki/index.md` 确定相关分类
2. 读该分类下的所有 sources/ 页面
3. 读相关的 concepts/ 和 entities/ 页面
4. 读已有的 comparisons/ 页面
5. 读已有的 syntheses/ 页面（避免重复工作）
6. 跟踪所有交叉引用，确保没有遗漏

**覆盖度检查：**
- 是否覆盖了所有相关的 sources/
- 是否覆盖了所有相关的 concepts/
- 是否覆盖了所有相关的 entities/
- 是否覆盖了已有的 comparisons/
- 是否有遗漏的交叉引用

#### Step 2 — 综合分析

**共识识别：**
- 如果多个独立来源支持同一观点 → 标注为"多源共识"
- 如果只有一个来源提到某个观点 → 标注为"单一来源"
- 如果多个来源给出不同数据 → 列出所有数据，不做平均

**矛盾处理：**
- 在报告中标注 `[!contradiction]`
- 列出矛盾双方的具体内容
- 不做调和，不做主观判断
- 建议用户提供更多资料来判断

**知识空白识别：**
- 哪些主题应该有信息但 Wiki 中没有
- 哪些概念被频繁引用但没有概念页
- 哪些对比有价值但还没有做

#### Step 3 — 撰写报告

**报告结构模板：**

```markdown
---
title: 主题 — 综合报告
type: synthesis
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [标签]
---

# [主题] — 综合报告

## 概述

[简要说明本报告的主题、范围和目的。2-3 句话]

## 各维度分析

### 维度一：[名称]

[分析内容。每个维度包含：定义/背景、关键发现、来源引用]

**来源：** [[wiki/sources/来源页.md]]

### 维度二：[名称]

[分析内容]

**来源：** [[wiki/sources/来源页.md]]

## 共识与分歧

| 观点 | 支持来源 | 证据强度 |
|------|---------|---------|
| [观点一] | [[来源1]], [[来源2]] | 多源共识 |
| [观点二] | [[来源3]] | 单一来源 |
| [观点三] | [[来源4]] vs [[来源5]] | [!contradiction] |

## 知识空白与建议

| 缺失内容 | 建议补充的资料类型 |
|---------|------------------|
| [缺失的知识点] | [建议补充什么类型的资料] |
| [缺失的知识点] | [建议补充什么类型的资料] |

## 参考文献

- [[wiki/sources/来源页.md]]
- [[wiki/concepts/概念页.md]]
- [[wiki/entities/实体页.md]]
- [[wiki/comparisons/对比页.md]]
```

#### 边界情况处理

| 情况 | 处理方式 |
|------|---------|
| **相关页面少于 3 个** | 告知用户"当前知识库中关于 X 的资料较少（仅 X 个来源），综合报告可能不够全面。建议先补充来源再综合" |
| **多个来源严重矛盾** | 在报告中突出矛盾，不做调和。建议用户提供更多资料来判断 |
| **主题跨多个领域** | 按领域分节，明确标注每个领域的覆盖程度（"本领域有 X 个来源"） |
| **用户要求更新已有综合报告** | 在原报告基础上追加新内容，标注"更新于 YYYY-MM-DD"。不删除原有内容 |
| **用户要求删除已有综合报告** | 不在 Wiki 中删除，标注"已废弃"或"已被 YYYY-MM-DD 版本替代" |
| **用户要求特定格式** | 在标准模板基础上，按用户要求调整 |
| **报告过长（>50KB）** | 拆分为多个子报告，用交叉引用连接 |

### 3.4 Lint（健康检查）

**触发条件：** 用户说"检查知识库"、"lint"、"健康检查"、"health check"、"clean up the knowledge base"等。

#### 检查清单及执行方法

##### 1. 矛盾检查

**执行方法：**
- 逐个读取 wiki/ 下的所有页面
- 对比同一主题的不同页面，看是否存在相互矛盾的陈述
- 特别关注：不同 sources/ 对同一事实的描述、concepts/ 的定义是否一致

**输出格式：**
```
| 页面 | 矛盾内容 | 涉及来源 |
|-----|---------|---------|
| pageA.md | 说 X | source1.md |
| pageB.md | 说 Y（与 X 矛盾） | source2.md |
```

##### 2. 孤立页面

**执行方法：**
- 扫描 wiki/ 下所有页面
- 对每个页面，搜索整个 wiki/ 目录中是否有其他页面引用它
- 被 0 个其他页面引用 → 孤立页面

**处理决策：**
- 孤立页面是否需要保留：如果内容有价值，创建入站引用
- 如果内容无价值，建议用户删除或合并

##### 3. 过时内容

**执行方法：**
- 检查每个页面的 updated 字段
- 对比同一主题的不同 sources/，看是否有更新的资料 supersede 了旧资料
- 特别关注：标准版本更新（如 GB 175-2023 supersede GB 175-2007）

**标注方式：**
- 在过时内容旁标注 `[!outdated] 已被 [[wiki/sources/新来源]] supersede`

##### 4. 缺失概念

**执行方法：**
- 扫描所有页面中的 `[[wiki/concepts/xxx]]` 引用
- 检查对应的概念页是否存在
- 扫描页面中提到的术语，看是否应该有对应的概念页

##### 5. 交叉引用

**执行方法：**
- 扫描所有页面中的 `[[wiki/...]]` 引用
- 检查每个引用指向的文件是否存在
- 注意：引用路径必须完整且正确

**死链接修复：**
- 如果目标文件被移动了 → 更新引用路径
- 如果目标文件被删除了 → 删除引用或替换为其他引用
- 如果目标文件不存在但应该存在 → 建议创建

##### 6. 格式一致性

**检查项：**
- YAML frontmatter 是否完整（每个页面都有？字段齐全？）
- type 字段是否使用了正确的值
- tags 是否一致（同一类型的页面使用相同的标签体系）
- 命名规范是否统一（同一类型的页面命名风格一致）
- 交叉引用格式是否正确（`[[wiki/...]]` 而不是 `[wiki/...]`）

##### 7. 知识空白

**执行方法：**
- 基于已有内容，判断哪些相关主题应该被覆盖但还没有
- 基于 sources/ 中的"建议补充"记录
- 基于用户之前的查询记录

#### 输出格式

**诊断报告** 保存到 `wiki/syntheses/lint-YYYY-MM-DD.md`

```markdown
---
title: 知识库健康检查报告 — YYYY-MM-DD
type: synthesis
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [lint, 健康检查]
---

# 知识库健康检查报告 — YYYY-MM-DD

## 概览

| 指标 | 数值 |
|------|------|
| 总页面数 | X |
| 发现问题数 | X |
| 严重问题 | X |
| 轻微问题 | X |

## 检查结果

### 1. 矛盾（X 处）

| 页面 | 矛盾内容 | 涉及来源 |
|-----|---------|---------|
| ... | ... | ... |

### 2. 孤立页面（X 个）

- [页面名] — 建议：[保留/合并/删除]

### 3. 过时内容（X 处）

- [页面名] — 内容：[描述] — 被 [[新来源]] supersede

### 4. 缺失概念（X 个）

- [概念名] — 被 [页面] 引用但无对应概念页 — 建议创建

### 5. 死链接（X 处）

| 来源页面 | 死链接 | 建议 |
|---------|-------|------|
| ... | [[wiki/...]] | 修复/删除/创建 |

### 6. 格式问题（X 处）

- [页面名] — 问题：[缺少 YAML / type 错误 / 标签不一致 / 命名不规范]

### 7. 知识空白（X 处）

- [主题] — 已有 X 个来源，但缺少 [具体内容]

## 修复建议

按优先级列出：
1. [紧急修复项]
2. [建议修复项]
3. [后续优化项]

## 补充资料建议

- [建议用户补充哪些来源]
```

#### 修复流程

1. **先报告，后修复** — 输出诊断报告，等用户确认后再执行修复
2. **重大修改先讨论** — 合并页面、删除页面等操作先与用户讨论
3. **修复后提交** — 修复完成后 Git 提交，commit message: `[Lint] YYYY-MM-DD 健康检查修复`

#### 修复优先级

| 优先级 | 问题类型 | 处理方式 |
|-------|---------|---------|
| P0（立即修复） | 死链接、格式错误 | 直接修复，不需要用户确认 |
| P1（建议修复） | 矛盾、过时内容 | 报告后等用户确认再修复 |
| P2（后续优化） | 孤立页面、缺失概念、知识空白 | 报告后等用户确认再处理 |

## 四、页面规范

### 4.1 YAML Frontmatter 完整模板

#### 通用页面（sources / concepts / entities / comparisons / syntheses）

```yaml
---
title: 页面标题
type: source | concept | entity | comparison | synthesis
```

## 五、Git 工作流

### 5.1 每次操作后必须提交

```bash
git add -A
git commit -m "[操作类型] 描述"
git push
```

**提交前检查：**
- [ ] 所有修改的文件都已保存
- [ ] 没有遗留的未跟踪文件（`git status` 确认）
- [ ] commit message 符合格式要求
- [ ] push 成功后确认远程已更新

### 5.2 提交信息格式

| 操作类型 | 格式 | 示例 |
|---------|------|------|
| Ingest | `[Ingest] 资料标题` | `[Ingest] 2026-06-20 国家标准资料集` |
| Synthesize | `[Synthesize] 主题名称` | `[Synthesize] 技术指标知识图谱` |
| Lint | `[Lint] YYYY-MM-DD 健康检查` | `[Lint] 2026-06-22 健康检查` |
| 其他更新 | `[Sync] 描述` | `[Sync] 新增 KNOWLEDGE-VAULT-LOGIC.md` |

### 5.3 Token 管理

- Token 存储在 Windows 凭据管理器中
- 使用 `credential.helper = manager` 自动获取
- **永不**将 token 硬编码在代码或文件中
- 如果 token 泄露，立即在 GitHub 设置中 revoke 并重新生成
- 如果 push 失败，检查：
  1. 网络连接（`github.com:443` 是否可达）
  2. Token 是否过期
  3. 远程分支是否有 upstream 配置

### 5.4 Push 失败处理

| 错误 | 原因 | 处理方式 |
|------|------|---------|
| `no upstream branch` | 分支未设置 upstream | `git push --set-upstream origin master` |
| `403/401` | Token 过期 | 从凭据管理器获取新 token 并更新 |
| `secret scanning` | 历史中包含敏感信息 | 用 `git filter-branch` 清除后重新 push |
| 连接超时 | 网络限制 | 检查网络，重试或换用 SSH |
| `non-fast-forward` | 远程有本地没有的提交 | 先 `git pull --rebase` 再 push |
| `error: failed to push` | 文件过大 | 检查是否有大文件，考虑使用 Git LFS |

### 5.5 提交频率规则

| 场景 | 提交时机 |
|------|---------|
| 单次 Ingest 操作 | 完成后立即提交 |
| 单次 Synthesize 操作 | 完成后立即提交 |
| 单次 Lint 操作 | 修复完成后提交 |
| 多次连续操作 | 每次操作独立提交，不合并 |
| 操作中途中断 | 已完成的修改提交，未完成的丢弃 |

## 六、决策树与边界情况

### 6.1 操作选择决策树

```
用户说了一句话
    │
    ├─ 包含触发词 → 执行对应操作
    │   ├─ "保存/摄入/ingest" → Ingest
    │   ├─ "查/知识库/wiki" → Query
    │   ├─ "综述/综合/synthesize" → Synthesize
    │   └─ "检查/lint/健康" → Lint
    │
    ├─ 包含多个触发词 → 按优先级：Lint > Ingest > Synthesize > Query
    │   （Lint 优先是因为修复问题后再操作更准确）
    │
    └─ 不包含触发词 → 正常对话，不操作知识库
```

### 6.2 通用边界情况处理

| 情况              | 处理方式                                 |                  |
| --------------- | ------------------------------------ | ---------------- |
| **操作过程中发现矛盾**   | 标注 `[!contradiction]`，继续操作，在 log 中记录 |                  |
| **操作过程中发现死链接**  | 修复死链接，在 log 中记录                      |                  |
| **用户中断操作**      | 已完成的修改提交，未完成的丢弃                      |                  |
| **多个用户同时操作**    | 每次操作前先 git pull，操作后 git push，冲突时手动解决 |                  |
| **Git push 失败** | 不丢失本地修改，修复问题后重试                      |                  |
| **页面数量过大**      | 单次操作控制在 50 个文件以内，超过时分批提交             |                  |
| **文件编码问题**      | 统一使用 UTF-8 编码                        |                  |
| **文件名包含特殊字符**   | Windows 不允许文件名中有 `/\\:*?"<>|`。标准编号中的 `/`（如 GB/T）必须替换为 `-`（如 GB-T）。已存在的 `GB\\T` 目录是错误，应合并到 `GB-T` |
| **跨平台换行符**      | Git 自动处理 CRLF/LF 转换，不手动干预            |                  |

### 6.3 质量门禁

每次操作完成前，检查以下条目：

- [ ] 所有新建页面都有 YAML frontmatter
- [ ] 所有引用都指向存在的页面
- [ ] 矛盾已标注 `[!contradiction]`
- [ ] index.md 已更新
- [ ] log.md 已追加
- [ ] Git 已提交并推送
- [ ] 远程仓库已确认更新

### 6.4 操作前的准备检查

每次操作开始前，检查以下条目：

- [ ] 已读 `wiki/index.md` 了解全貌
- [ ] 已读 `wiki/hot.md` 了解近期上下文
- [ ] 本地仓库是最新状态（`git pull`）
- [ ] 没有未完成的修改（`git status` 确认）
- [ ] 了解本次操作的目标和范围

## 七、注意事项

### 7.1 通用规则
- 先读 `wiki/index.md` 了解全貌，再执行具体操作
- 保持所有页面格式一致
- 交叉引用双向维护（A 引用 B 时，B 也应有回链）
- 操作完成后更新 `wiki/log.md`
- 每个标准节点页面包含"引用标准"和"关联标准"部分
- 不要一次性创建超过 50 个文件
- 操作完成后检查远程仓库是否同步

### 7.2 安全规则
- 不修改 raw/ 下的原始文件
- 不在代码/文件中硬编码敏感信息（token、密码等）
- 遵守 GitHub secret scanning 规则
- 不摄入包含个人隐私的资料
- 如果 token 泄露，立即在 GitHub 设置中 revoke
- 不使用 `git push --force` 除非明确需要覆盖远程历史

### 7.3 质量规则
- 每个页面必须有 YAML frontmatter
- 引用必须指向存在的页面
- 矛盾必须标注 `[!contradiction]`
- 知识不足时明确告知，不编造
- 综合报告不是简单拼接，而是真正的跨源综合
- 指标值的 YAML frontmatter 必须包含 constraint、value、unit、grade 四个字段
- 同一类型的页面使用相同的标签体系
- 文件名中的空格和特殊字符保持原样（如指标值文件）

### 7.4 关于 standards/ 模块的特别规则
- 新建标准时，必须同时创建 L1-L4 所有层级
- 同一指标值被多个标准共享时，优先使用已有的 L4 文件
- 指标值的 grade 字段：无等级区分时填"通用"
- 产品类别名称必须与监管目录一致（工业生产资料、农业生产资料等）
- 标准编号中的连字符：强制性标准用 `GB`，推荐性标准用 `GB`/`T`（对应标准原文的 `/`）
- 标准节点页面的"引用标准"部分列出该标准引用的其他标准编号和名称
- 标准节点页面的"关联标准"部分列出与本标准相关的其他标准（用 Wiki 链接）

### 7.5 关于 hot.md 的使用规则

`wiki/hot.md` 是当前会话的热缓存，用于记录本次会话的上下文。

**写入时机：**
- 会话开始时，记录本次会话的目标
- 每次操作后，记录操作摘要
- 会话结束时，清空或保留关键信息

**格式：**
```markdown
# Hot Cache — 当前会话

## 会话目标
[本次会话要完成的任务]

## 操作记录
- [操作时间] [操作类型] [摘要]

## 注意事项
[本次会话需要记住的上下文]
```

**规则：**
- hot.md 的内容是临时的，不会被 Git 提交
- 每次会话开始时读取，结束时清空
- 如果 hot.md 中记录了有价值的信息，迁移到 log.md 或对应的 Wiki 页面
